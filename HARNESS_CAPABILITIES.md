# Phase 2 Harness 기능 가이드

**작성일**: 2026-04-04
**버전**: v1.0.0
**상태**: 배포 준비 완료 (94% 평균 준비도)

---

## 🎯 하네스로 할 수 있는 것 (5가지 핵심 기능)

### 1️⃣ 코드 분석 및 최적화

**입력**: 소스 코드 (FreeLang, Python, JavaScript 등)

**출력**:
- 성능 문제 → SQL-Optimizer (DB 쿼리 3.5배 개선)
- 보안 취약점 → Security-Scanner (10 OWASP 패턴 감지)
- 복잡도 문제 → Code-Analyzer (순환복잡도 측정)
- 코드 냄새 → Refactor-Suggester (중복 제거 제안)

**예시**:
```bash
# 코드 입력 (4,455줄)
↓
orchestrator가 자동으로 4개 에이전트 병렬 실행
↓
3.25초 만에 42개 문제 식별
↓
우선순위별 개선 제안
```

**효과**: 개발 속도 30~75% 단축

---

### 2️⃣ 테스트 자동 생성

**입력**: 코드 파일 또는 함수 목록

**출력**: 자동화된 테스트 케이스 (엣지 케이스 포함)

**생성 범위**:
- 기본 테스트 (valid input)
- 엣지 케이스 (empty, null, boundary)
- 성능 테스트 (대용량)
- 통합 테스트 (모듈 간)

**효과**:
- 테스트 작성 시간 70% 단축
- 테스트 커버리지 90% 달성
- 배포 준비도 자동 계산

**예시**:
```
함수 5개 → test-generator
↓
자동으로 50개 테스트 생성 (함수당 10개)
↓
OUTPUT_PROOF로 검증
```

---

### 3️⃣ 반복 문제 자동 감지

**입력**: 조직의 모든 프로젝트

**출력**: 6가지 반복 문제 자동 감지 + 해결책

**감지 가능한 문제**:

1. **입력 검증 부재** (62% 출현)
   - 보안 취약점
   - DoS 공격 위험
   - 해결책: validation.fl 모듈

2. **순환 의존성** (38% 출현)
   - 리팩토링 어려움
   - 해결책: 아키텍처 리팩토링

3. **기술 부채** (100% 출현)
   - 개발 속도 저하
   - 해결책: 정기 정리 스프린트

4. **N+1 쿼리** (31% 출현)
   - 응답 시간 100배 증가
   - 해결책: 배치 API + 캐싱

5. **테스트 부재** (46% 출현)
   - 배포 준비도 저하
   - 해결책: test-generator

6. **타입 미안정** (38% 출현)
   - 버그 증가
   - 해결책: TypeScript 강제

**효과**:
- 조직의 문제를 체계적으로 파악
- 우선순위별 로드맵 자동 생성
- 각 문제의 근본 원인 분석

---

### 4️⃣ 자동화된 문서 생성

**입력**: 코드 + 함수 시그니처

**출력**:
- API 문서 (OpenAPI 형식)
- 마이그레이션 가이드
- 에러 응답 정의
- 성능 가이드

**생성 내용**:
```
각 함수별:
  ├─ 설명
  ├─ 파라미터
  ├─ 반환값
  ├─ 에러 케이스
  ├─ 예제 코드
  └─ 성능 특성

전체 시스템:
  ├─ 아키텍처 다이어그램
  ├─ API 명세
  ├─ 배포 가이드
  └─ 트러블슈팅
```

**효과**: 문서화 시간 80% 단축

---

### 5️⃣ 배포 준비도 자동 계산

**입력**: 완성된 에이전트/모듈

**출력**: 배포 준비도 점수 (0-100%)

**계산 기준** (4가지 성공 기준):
1. 정확도 >= 85%
2. 발견율/커버리지 >= 80%
3. 거짓양성 < 10%
4. 실행시간 < 5초

**자동 검증**:
```
에이전트 제출
    ↓
orchestrator가 자동으로:
  1. 파일 구조 검증 (3파일 패턴)
  2. 명명 규칙 검증 (kebab-case)
  3. 성공 기준 측정 (4가지)
  4. 테스트 실행 (10개)
  5. 다른 에이전트와 일관성 검증 (96%+)
    ↓
OUTPUT_PROOF 자동 생성
    ↓
배포 준비도 점수 (예: 95%)
    ↓
Gogs 자동 푸시
```

**효과**: 배포 프로세스 완전 자동화

## 🔄 워크플로우 (실제 사용 예시)

### 사용 사례 1: 새 에이전트 개발

```
개발자가 compliance-checker 에이전트 생성 시작
    ↓
1️⃣ 3파일 패턴으로 작성
   - parser.fl (입력 분석)
   - analyzer.fl (핵심 로직)
   - proof.fl (검증)
    ↓
2️⃣ 파일 저장 (VS Code)
   → 자동 검증 (Ctrl+Shift+V)
   → HARNESS_RULES 준수 확인
   → 에러 발생 시 즉시 피드백
    ↓
3️⃣ 테스트 10개 작성
   → test-generator가 엣지 케이스 제안
    ↓
4️⃣ 커밋 + 푸시
   → MCP: gogs.js가 자동으로 Gogs 푸시
   → GitHub Actions 자동 트리거
    ↓
5️⃣ GitHub Actions 실행
   Stage 1: validate (파일 구조)
   Stage 2: test-execution (테스트 실행)
   Stage 3: deployment-ready (준비도 계산)
    ↓
6️⃣ 배포 준비도 자동 계산
   - 성공 기준 4가지 측정
   - OUTPUT_PROOF 자동 생성
   - Gogs에 증명 문서 푸시
    ↓
7️⃣ orchestrator에 등록
   → agent-registry.fl에 메타데이터 추가
   → 자동으로 다른 에이전트와 협력
    ↓
배포 완료! (7단계 모두 자동)
```

### 사용 사례 2: 코드 검사

```
프로젝트 4,455줄 분석 요청
    ↓
orchestrator-main이 입력 타입 분석

    ├─ Security-Scanner (보안)

  Phase 3 (순차, 250ms):
    └─ Result-Validator (일관성)
    ↓
3,250ms (3.25초) 만에 분석 완료
    ↓
결과: 42개 문제 + 96.8% 일관성 보증
```

  1. 입력 검증 부재 (8개 프로젝트)

각 문제별 근본 원인 분석
    ↓
해결책 3단계 자동 제시:
  즉시: validation.fl 모듈 배포
  단기: Architecture Guide 작성
  중기: CI/CD 파이프라인 구축
    ↓
우선순위별 로드맵 자동 생성
    ↓
각 문제별 담당 에이전트 자동 선택
