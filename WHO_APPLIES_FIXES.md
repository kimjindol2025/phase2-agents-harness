# 개선은 누가 하나? (누가 패치를 적용하나?)

**작성일**: 2026-04-04
**상황**: 현실적 문제 분석 및 해결책

---

## 🤔 현재 상황 (문제)

### 자동 분석 ✅ vs 수동 적용 ❌

```
orchestrator의 분석
    ↓
pattern-detector: 6가지 문제 감지 ✅
code-analyzer: 복잡도 분석 ✅
security-scanner: 보안 패치 생성 ✅
test-generator: 테스트 생성 ✅
    ↓
자동 생성 파일:
  ├─ security.patch ✅
  ├─ tests.fl ✅
  ├─ ANALYSIS_REPORT.md ✅
  └─ REFACTORING_PLAN.md ✅
    ↓
그런데... 누가 이걸 적용함?
    ↓
각 프로젝트의 개발자가 수동으로:
  $ git apply security.patch
  $ git commit -m "..."
  $ git push
    ↓
문제:
  1. 개발자가 바쁘면 미루어짐
  2. 우선순위가 낮으면 무시됨
  3. 각 팀마다 진행도가 다름
  4. 추적 불가능
  5. 자동화라는 취지 상실
```

## 💡 해결책 (3가지 방식)

### 방식 1: 개발자 중심 (현재)

```
┌─────────────────────────────────────┐
│ 자동 분석 & 패치 생성               │
│ (하네스가 함)                       │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 개발자가 검토 & 승인 & 적용         │
│ (개발자가 직접 함)                  │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 기본 제공 형태: 옵션                 │
│ "좋은 패치니까 써도 좋아"           │
└─────────────────────────────────────┘

장점: 개발자가 최종 통제권
단점: 자동화 효율 감소, 추적 어려움
```

### 방식 2: 자동 적용 (권장)

```
┌─────────────────────────────────────┐
│ 자동 분석 & 패치 생성               │
│ (하네스가 함)                       │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 자동으로 Pull Request 생성          │
│ PR 제목: "fix: [AUTO] 입력 검증"   │
│ 설명: ANALYSIS_REPORT 포함          │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ CI/CD 자동 검증                     │
│ ✓ 파일 구조 검증                    │
│ ✓ 테스트 실행                       │
│ ✓ 린트 검사                         │
│ ✓ 성공하면 자동 병합                │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Gogs 자동 푸시 & 배포               │
│ (모두 자동)                         │
└─────────────────────────────────────┘

장점: 완전 자동, 빠름, 추적 가능
단점: 개발자의 통제권 감소 (하지만 PR 검토 가능)
```

### 방식 3: 하이브리드 (안전)

```
패치 분류:
  ├─ 자동 적용 (위험도 낮음, 테스트 통과)
  │   ├─ validate_input() 추가
  │   ├─ 테스트 케이스 추가
  │   └─ 문서화 추가
  │   → 자동 PR + 자동 병합
  │
  └─ 수동 검토 (위험도 높음, 복잡한 리팩토링)
      ├─ 순환 의존성 제거
      ├─ 아키텍처 변경
      └─ 성능 최적화
      → 자동 PR + 개발자 수동 병합

위험도 판단: 테스트 통과도 + 변경 라인 + 복잡도
```

## 🤖 새로운 에이전트: auto-applier (자동 적용기)

### 개념

```
pattern-detector: 문제 감지
    ↓
security-scanner: 패치 생성
    ↓
auto-applier: ← 새 에이전트!
    ├─ 패치 검증 (위험도 판단)
    ├─ PR 자동 생성
    ├─ CI/CD 자동 검증
    ├─ 병합 자동화 (안전한 패치)
    └─ 개발자 알림 (위험한 패치)
    ↓
Gogs 자동 푸시
```

```

├─ 위험도: 낮음
├─ 테스트: 자동으로 추가된 테스트
├─ 통과율: 100%
├─ 자동 PR 생성
├─ CI/CD 검증 (2분)
├─ 자동 병합 ✅
└─ Gogs 푸시

문서화 추가
├─ 위험도: 아주 낮음
├─ 테스트: 필요 없음
├─ 자동 PR 생성
├─ 자동 병합 ✅

```

├─ PR 자동 생성
├─ 개발자 검토 요청 (PR comment)

└─ Gogs 푸시

├─ 위험도: 중간-높음
├─ 변경 라인: 30-100
├─ 성능 테스트 필수
├─ PR 자동 생성
├─ CI/CD에서 성능 벤치마크 실행

├─ changes_line < 50
└─ 만족하면 자동 병합 + PR close + Gogs 푸시
