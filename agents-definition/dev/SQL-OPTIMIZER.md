- **분류**: 개발 에이전트 (dev)
- **목표**: gogs-server-fl의 느린 쿼리를 분석하고 최적화 제안
- **우선순위**: 🔴 Phase 1 (2026-04-16 배포 목표)
- **상태**: 설계 중

---

## 담당 업무

### 입력 (Input)
```
1. 느린 쿼리 로그 (source: gogs-server-fl)
   - SQL 문장
   - 실행 시간 (ms)
   - 행(row) 수
   - 락 타입 (READ/WRITE)

2. 테이블 스키마 (gogs-server-fl DB)
   - repositories
   - commits
   - branches
   - refs
   - ...

3. 벤치마크 조건
   - 동시 요청 수: 1K, 5K, 10K
   - 데이터 크기: 1M~100M rows
```

### 처리 로직
```
Step 1: 쿼리 분석 (AI + 구조)
  - EXPLAIN QUERY PLAN 실행
  - JOIN 순서 최적화 여부 검사
  - 인덱스 사용 여부 검사
  - 서브쿼리 최적화 가능성 검사

Step 2: 최적화 전략 수립 (AI)
  - 인덱스 추천 (컬럼 조합 및 순서)
  - 쿼리 리팩토링 (JOIN 순서 변경 등)
  - 트랜잭션 설계 (배치 처리, PRAGMA 튜닝)
  - 읽기 캐시 도입 제안

Step 3: 최적화 코드 생성 (AI)
  - 개선된 SQL 문장 작성
  - 인덱스 생성 DDL 작성
  - 캐시 레이어 코드 생성 (FreeLang v4)

Step 4: 검증 (구조)
  - 벤치마크 환경에서 실행
  - before/after 성능 비교
  - 정확도 검증 (결과 행 수 동일 여부)
```

### 출력 (Output)
```
1. 최적화 리포트 (Markdown)
   - 발견 사항 요약
   - 병목 지점 분석
   - 추천 최적화 (우선순위 순)
   - 예상 성능 개선율

2. 최적화 코드
   - 개선된 SQL (sql-optimizer/optimized-queries.sql)
   - 인덱스 DDL (sql-optimizer/indexes.sql)
   - FreeLang 캐시 레이어 (sql-optimizer/cache-layer.free)

3. 벤치마크 결과
   - 동시성 테스트 결과 (1K/5K/10K)
   - Req/sec 비교 (before/after)
   - 응답시간 분포 (p50, p95, p99)
   - 메모리 사용량
```

## 성공 기준

| 기준 | 목표 | 검증 방법 |
|------|------|---------|
| **성능 향상** | 10K 동시 요청에서 Req/sec 10배 이상 | 벤치마크 테스트 |
| **정확도** | 최적화 쿼리 결과 = 원본 쿼리 결과 | 행(row) 수 동일 + 샘플 비교 |
| **안정성** | 메모리 누수 없음, 데드락 없음 | 1시간 연속 테스트 |
| **적용 가능성** | 3개 이상의 느린 쿼리 최적화 적용 | gogs-server-fl 프로덕션 테스트 |

## 에이전트 한계점 & 보완 구조

### AI 한계점
- ❌ 복잡한 동시성 문제 자동 분석 불가 (메모리 누수, 데드락)
- ❌ 실시간 메트릭 변화에 따른 동적 인덱싱 제안 어려움
- ❌ 특정 데이터 분포에 최적화된 쿼리 플랜 생성 불가

### 보완 구조 (프롬프트 체인 + 외부 도구)
1. **SQL 분석**: 자체 EXPLAIN PLAN 실행 (구조)
2. **인덱스 추천**: 통계 기반 알고리즘 (구조)
3. **성능 검증**: 자동 벤치마크 (구조)
4. **최종 판단**: AI 재검토 (AI)
